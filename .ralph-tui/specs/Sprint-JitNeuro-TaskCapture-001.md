## Problem

You are in a deep Claude Code session and think of a task. Current options:
- Type it in the current session -- pollutes context, gets mixed into active work
- Open a separate session -- requires context switch, must remember what you were doing
- Write it somewhere manually -- breaks flow, often forgotten

None of these work well for maintaining focus. The task either disrupts current work or gets lost.

---

## Goal

Zero-friction task capture from ANY Claude Code session, without loading task context into
the current session. Paired with a visual dashboard so you can scan current state in 5
seconds without holding the list in your head.

---

## Options Evaluated

### Option A: /task JitNeuro Command (SELECTED FOR PHASE 1)
New workspace-level command at .claude/commands/task.md

Usage: `/task "description"` or `/task "description" --project jitai --priority high`

Behavior:
- Appends task to .claude/tasks/pending.md ONLY
- Does NOT load existing tasks or context into the current session
- Optional: if JitTask MCP is configured and reachable, also creates ADO work item
- Confirms append in 1 line: "Task added: [description]" -- no more output
- A team member or scheduled job reviews pending.md and moves to your task board

Why: Zero context pollution. One command. Works from any session. No window switching.

### Option B: Dedicated task session
Always-open Claude Code window in a separate task manager repo.

REJECTED for daily use. Requires context switching to add a task, which means you must
remember what you were doing when you tab back. This is the opposite of focus optimization.
Acceptable as a secondary interface but not the primary task capture path.

### Option C: TUI/Web Dashboard (SELECTED FOR PHASE 2)
Visual panel showing: active tasks by priority, current sprints, NEEDS DAN items, and
buttons to add a task or kick off a Claude skill.

Options for implementation:
- C1: Extend ralph-tui (terminal, already exists, no new infra)
- C2: New page in jitai admin dashboard (web, always accessible in browser)
- C3: Standalone n8n dashboard (connects to ADO data, no new app needed)
- C4: VS Code extension or Claude Code status line integration

C2 (jitai admin page) is recommended -- same stack (Next.js/React), always-visible in
browser alongside Claude Code, and can be extended to launch Claude skills via API calls.
Pairs with the blog admin dashboard sprint (Sprint-Blog-Admin-001) for shared UI patterns.

## Phase 1: /task Command (Sprint-JitNeuro-TaskCapture-001)

### US-001: Create /task workspace command
**File:** .claude/commands/task.md
**Behavior:**
- Accepts: task description (required), --project (optional), --priority (optional: low/medium/high)
- Appends to .claude/tasks/pending.md with timestamp, session CWD, project tag
- Output: single confirmation line only
- Does NOT trigger context load, bundle load, or memory read
- Does NOT display existing tasks

**Format in pending.md:**
```
- [ ] [2026-03-12 14:32] [project: jitai] [priority: high] Fix the unpublish button
- [ ] [2026-03-12 15:01] [project: jitneuro] [priority: medium] Add /task command
```

**Pass:** `/task "Fix X"` appends to pending.md, single-line confirmation, no context change
**Fail:** Command loads task list into session context, or takes more than 2 seconds

### US-002: Create pending.md scaffold
**File:** .claude/tasks/pending.md
**Content:** Header only (project, date created, format spec). Empty task list.
**Pass:** File exists, writable from any workspace session
**Fail:** File missing, permissions issue

### US-003: Update /dashboard command to include pending tasks
**File:** .claude/commands/dashboard.md
**Change:** Add section reading pending.md and displaying unprocessed tasks at top of
dashboard output under "UNPROCESSED TASKS" heading.
**Pass:** /dashboard shows pending.md tasks
**Fail:** Tasks not visible or pending.md read errors crash dashboard

## Architectural Constraint (set 2026-03-12)
This solution lands in JitNeuro as a MODULE, not a one-off command set.

- JitNeuro commands (/task, /dashboard, etc.) call the module
- The module calls the configured backend via MCP
- Backend is swappable by config, not by rewriting commands
- v1 reference backends: GitHub Issues (gh CLI) + ADO Boards (JitTask MCP)
- Interface contract the module must fulfill: create / read / update / delete / list / prioritize
- Any future backend (Linear, Jira, plain markdown, etc.) implements the same interface

Discovery + interface spec must be completed and approved by the project owner BEFORE any build starts.
