## Problem Statement

JitNeuro v1.0 has per-repo team context (.jitneuro/) working for engrams and rules. But cross-project bundles (active-work, infrastructure, tech-stack) are stuck in .claude/bundles/ which is personal, gitignored, and unshareable. The architecture needs to support:
- Solo devs (no team)
- Small teams (2-5 people)
- Enterprise (segmented teams, different repo access)
- Cross-platform (Windows, Mac, Linux -- different paths)
- Same user, multiple machines (Dan: DevVM + DanS-Laptop)

**Critical:** .claude/ is NOT in a git repo. No push, no pull, no sync. Even a single developer with two machines cannot share cross-project bundles through .claude/. Anything team-shared MUST live in a git repo.

## The Registry Map: bundle-map.json

### What it is

A machine-specific JSON file at .claude/bundle-map.json that maps logical bundle/engram names to absolute file paths on THIS machine. Auto-generated by /onboard --all. Not manually edited.

### Why it exists

- Routing weights use logical names: [infrastructure] not full paths
- Each machine has different absolute paths (D:\Code vs ~/Code vs C:\Users\...)
- The map decouples "what to load" from "where it lives on this machine"
- Portable routing weights: same routing works on every machine

### Format

```json
{
  "generated": "2026-03-29T17:00:00Z",
  "generatedBy": "/onboard --all",
  "bundles": {
    "infrastructure": "D:/Code/.jitneuro/bundles/infrastructure.md",
    "tech-stack-core": "D:/Code/.jitneuro/bundles/tech-stack-core.md",
    "active-work": "D:/Code/.jitneuro/bundles/active-work.md",
    "aibm": "D:/Code/AIFS/ai-boat-mechanic/.jitneuro/bundles/aibm.md",
    "covenai": "D:/Code/Futures/CovenAI/.jitneuro/bundles/covenai.md",
    "blog-content": "D:/Code/.jitneuro/bundles/blog-content.md"
  },
  "engrams": {
    "jitneuro": "D:/Code/jitneuro/.jitneuro/engrams/jitneuro-context.md",
    "customers": "D:/Code/Customers/.jitneuro/engrams/customers-context.md",
    "aifs-core": "D:/Code/AIFS/AIFieldSupport-API/.jitneuro/engrams/aifs-core-context.md",
    "sqlbackup": "D:/Code/Tools/.jitneuro/engrams/sqlbackup-context.md"
  }
}
```

### Personal config stays in .claude/jitneuro.json (manually edited):

```json
{
  "machineName": "DevVM",
  "teamContextPaths": ["D:/Code/.jitneuro"]
}
```

Separation: jitneuro.json = settings you edit. bundle-map.json = registry the system builds.

### How it's built

/onboard --all scans in order:
1. Each repo's .jitneuro/bundles/*.md and .jitneuro/engrams/*.md
2. Each path in teamContextPaths: <path>/bundles/*.md and <path>/engrams/*.md
3. .claude/bundles/*.md and .claude/engrams/*.md (personal fallback)

Filename without .md = logical name. First discovered instance wins (repo > team > personal).

### Resolution order at runtime

When a routing weight says [infrastructure]:
```
Priority 1: Current repo's .jitneuro/bundles/infrastructure.md   (repo override)
Priority 2: bundle-map.json["bundles"]["infrastructure"]          (registered path)
Priority 3: .claude/bundles/infrastructure.md                     (personal fallback)
```

### Missing path behavior

- File missing at mapped path: skip silently during normal work (no nagging)
- Auto-recover on session start: re-validate map, detect restored files
- /health always flags: missing entries shown as WARN
- Smart enough to not prompt every time. Resilient enough to find it when restored.

## Team Segmentation

For teams with different access levels, use multiple team context repos:

```
D:\Code\.jitneuro\              <-- org-wide (everyone clones)
  bundles/infrastructure.md
  bundles/tech-stack-core.md

D:\Code\AIFS\.jitneuro-team\    <-- AIFS team only (restricted GitHub access)
  bundles/aifs-domain.md

D:\Code\marketing\.jitneuro\    <-- marketing team only
  bundles/content-strategy.md
```

Each developer lists only the repos they have access to in .claude/jitneuro.json:
```json
{
  "teamContextPaths": [
    "D:/Code/.jitneuro",
    "D:/Code/AIFS/.jitneuro-team"
  ]
}
```

Git permissions = access control. Can't clone = don't have the path = bundles invisible. No RBAC needed.

## How /learn Writes Updates

When /learn identifies a bundle update:
1. Classify: REPO-SPECIFIC, CROSS-PROJECT, or PERSONAL
2. Look up the target in bundle-map.json to find the file path
3. If REPO-SPECIFIC: write to current repo's .jitneuro/bundles/
4. If CROSS-PROJECT: write to the path in bundle-map (team context repo)
5. If PERSONAL: write to .claude/bundles/
6. After writing, commit to the team repo (the bundle is in a git repo)

## How /save Writes active-work

active-work.md is a cross-project bundle. /save writes it to bundle-map["active-work"] which points to the team context repo. Team members pull to see latest status.

## Solo User Experience

- teamContextPaths absent or empty
- bundle-map.json maps everything to .claude/bundles/ (personal)
- /onboard --all builds the map from .claude/ paths
- Zero change from v0.x behavior

## Install/Setup Flow

**Solo:**
1. Run install.ps1 -- copies templates to .claude/
2. /onboard --all -- scans repos, builds engrams, builds bundle-map

**Team:**
1. Run install.ps1 -- copies templates to .claude/
2. Clone the team context repo(s)
3. Set teamContextPaths in .claude/jitneuro.json
4. /onboard --all -- scans all repos + teamContextPaths, builds full bundle-map
5. Cross-project bundles resolve from team context repo
6. git pull on team context repo to get updates

## Security Rule (HARD)

ALL engrams and bundles MUST reside in a git repo's .jitneuro/ folder.
If a user does not have access to clone the repo, they do not see its context.
Git permissions = context permissions. No exceptions.

This means:
- .claude/bundles/ holds ONLY personal drafts (not production bundles)
- .claude/engrams/ holds NOTHING in production (all in repos)
- The bundle-map only contains paths inside repos the user has cloned
- /onboard --all only indexes repos that exist on disk

## Active-Work Architecture (Divergent Analysis Result)

Evaluated 4 approaches: Per-User (A), Domain Split (B), Hybrid (C), Append-Log (D).
Converged on C (Hybrid Domain Split + Per-User Overlay).

### Problem
Single active-work.md (128 lines) updated by /save every session. Two devs + two machines = merge conflicts guaranteed. Highest write-frequency file in the system.

### Solution: Approach C -- Domain Split + Per-User Overlay

active-work.md transforms from 128-line monolith to 15-line index with descriptions:

```
<workspace>/.jitneuro/bundles/
  active-work.md              <-- index with descriptions (15-25 lines)
  active-work/
    decisions.md              <-- locked decisions, archived sprints (rarely changes)
    needs-owner.md            <-- per-author sections (append-only, git auto-merges)
    blog.md                   <-- blog content + platform status
    inbound-agency.md         <-- inbound marketing, N8N, email
    jit-dash.md               <-- dashboard status
    aibm.md                   <-- AIBM status
    jitpub.md                 <-- JitPub + JitSage
    jitneuro-marketing.md     <-- CovenAI + YouTube series
```

active-work.md index format (with descriptions for human + AI):
```markdown
# Bundle: Active Work

## Current Sprints
- AIFieldSupport Response Speed: IN PROGRESS
- jitneuro (DOE v2): DEPLOYED

## Domain Status
| Domain | File | Description | Updated |
|--------|------|-------------|---------|
| Blog | active-work/blog.md | Ghost pipeline, content quality, 8 drafts pending | 2026-03-29 |
| Inbound | active-work/inbound-agency.md | N8N workflows, marketing, email, assessments | 2026-03-29 |
| jit-dash | active-work/jit-dash.md | Task board, content review dashboard | 2026-03-29 |
```

Per-user files stay exactly as they are (already implemented):
```
<repo>/.jitneuro/users/
  Dan Stolts/DevVM/active-work.md      <-- what Dan is doing on DevVM right now
  Dan Stolts/DanS-Laptop/active-work.md
  tristan/active-work.md               <-- what Tristan is doing
```

### Conflict Analysis

| Scenario | Conflict? | Why |
|----------|-----------|-----|
| Dan edits blog, Tristan edits jit-dash | NO | Different domain files |
| Dan on DevVM edits blog, Dan on Laptop edits blog | YES (meaningful) | Same domain |
| Sprint handoff (Dan finishes, Tristan picks up) | NO | Domain file stays, no copy/move |
| Both devs on same sprint | OK | Same domain, meaningful conflict if concurrent |
| NEEDS OWNER items from both devs | NO | Per-author sections, git auto-merges |

### needs-owner.md Format (per-repo in Hub.md, aggregated by /health)

NEEDS OWNER items live per-repo in Hub.md (source of truth). /health and /sessions aggregate at runtime. No separate cross-repo needs-owner file.

```
Per-repo (source of truth):
  <repo>/.HUB/Hub.md               "BLOCKED: push approval needed"
  <repo>/.jitneuro/users/<name>/   "Blockers: waiting on API key"

Cross-project (aggregated at runtime):
  /sessions dashboard               reads all Hub.md, compiles NEEDS OWNER
  /health                           surfaces NEEDS OWNER as part of health check
```

### Scaling

- 2-3 devs: zero conflicts on different domains
- 10 devs: domains get 2-3 concurrent editors max
- 50+ devs: sub-domain splits supported (active-work/inbound-agency/content.md)
- Domains grow with the business, not headcount
